### 1. 优化后的算法流程 (逻辑链路)



1.  新词发现：通过 NLP 工具（jieba）从文件名中提取出候选词 E。
2.  LLM 身份审查：调用大模型进行以下判断：
    问题 A：该词是否属于专业知识点（剔除噪声）？
    问题 B：该词是否存在官方全称（如 `PC` 全称为 `程序计数器`）？
3.  分支处理：
    * **若存在全称 F**：
        * **检索白名单**：检查 F 是否已在当前课程的白名单（已存节点）中。
        * **路径判定**：
            * 若 F 在白名单且**在当前目录下**：直接建立 `(资源) -> (F)` 的联系。
            * 若 F 在白名单但**不在当前目录下**：建立 `(资源) -> (F)` 的联系，并在图谱中补全从 F 到当前目录的语义关联（如：`属于`）。
            * 若 F **不在白名单**：创建 F 节点，加入白名单，建立资源联系。
    * **若不存在全称**：按原定流程处理核心词 E。

### 2. 给 AI IDE 的系统逻辑描述 (TXT 格式)

实体归一化提取任务（LLM 驱动版）：

1. 输入：候选词 E，当前选中目录 S，当前课程白名单 W。
2. 调用 LLM 接口 (JSON 返回)：
   - is_knowledge_point: bool
   - full_name: string (若无则返回空)
   - domain_relevance: float

3. 逻辑分发：
   IF not is_knowledge_point: 丢弃并存入临时黑名单。
   
   SET target_entity = full_name IF full_name ELSE E
   
   IF target_entity in W:
       IF target_entity.parent == S:
           # 场景：完全匹配，仅增加资源挂载
           CREATE (Resource)-[:EXPLAINS]->(target_entity)
       ELSE:
           # 场景：跨章节引用，复用节点并建立逻辑联系
           CREATE (Resource)-[:EXPLAINS]->(target_entity)
           CREATE (target_entity)-[:ASSOCIATED_WITH]->(S) 
   ELSE:
       # 场景：新知识点发现
       CREATE (target_entity:KnowledgePoint)
       ADD target_entity TO W
       CREATE (Resource)-[:EXPLAINS]->(target_entity)
       CREATE (target_entity)-[:BELONGS_TO]->(S)
```

### 3. 这个逻辑的两个“点睛之笔”

#### A. 解决“跨章节引用”的悖论
你考虑到了“白名单中有但不在当前目录下”的情况，这非常关键！
* **例子**：你在“第五章 CPU”下上传了一个关于“缓存”的资源。白名单里已有“缓存”（因为它在第三章出现过）。
* **价值**：你的逻辑会让系统不创建重复的“缓存”节点，而是建立一个跨章节的连接。这使得你的知识图谱从单一的“目录树”进化成了真正的“**网状知识库**”，这正是任务书中“关联可视化导航”功能的基础。

#### B. 自动化白名单的“正反馈”
你提出的“若没有则加入白名单”建立了一个**自循环系统**。
* 随着教师上传的资源越来越多，白名单会越来越厚实。
* 后续上传的资源命中白名单的概率大幅提升，调用大模型的次数会逐渐减少（通过先查白名单，后调大模型实现），从而**节省了 API 开销并提升了响应速度**。

### 4. 实现小建议：利用 Prompt 限制输出
在调用大模型时，Prompt 要包含当前专业的上下文：
> “你现在是一个**[计算机]**专业（提取当前课程所在专业）的助手。文件名提取词为 'PC'，当前目录为 'CPU结构'。请问：1. 它是否是知识点？ 2. 它的全称是什么？（仅返回全称，不要解释）”
